Bug #2: November 2025 Stripe Incident

üö® CONFIRMED PUBLIC INCIDENT

November 2025: Multiple Android users charged $400‚Äì600+ via Stripe without authorization. Qwick acknowledged: "billing system was incorrectly configured and sent charges at an incorrect interval." Accounts suspended post-incident.

‚ö† BEFORE ‚Äî Dual Stripe Configuration

What happened (technical)

Two Stripe configurations ran simultaneously post-merger:

  • Qwick's original Stripe account (webhooks: charge.success ‚Üí qwick.com/webhooks)
  • Gigpro's Stripe account (webhooks: charge.success ‚Üí gigpro.com/webhooks)

Migrated users had payment tokens in both systems. Android users (primary Gigpro app bundle = com.gigpro) hit both webhook endpoints for a single transaction.

The fix

  • Audit all active Stripe webhook endpoints across both legacy configurations
  • Migrate all payment tokens to single Stripe account
  • Implement idempotency keys on all charge operations
  • Add charge deduplication layer (Redis or DB-level)
  • Archive (not delete) Gigpro Stripe configuration after verified migration

If both configurations are still active, every migrated Android user is at risk of re-occurrence. CFPB complaint threshold: 5+ unauthorized charges to same merchant = formal investigation trigger. One news article about "gig app stealing from hospitality workers" is brand-level damage.
