üîí Persistent Clock-In State WORKSTREAM 2 ‚Äî CRITICAL

The documented clock-out bug: Hyperview client re-fetches the shift screen after the scheduled end time, server returns an updated state, client renders no active shift. Worker cannot clock out. System marks no-show.

The fix: A local-first shift state layer that persists clock-in events to AsyncStorage (React Native) independent of HXML server responses. The clock-out button is rendered from local state, not HXML ‚Äî it only disappears after a confirmed server clock-out acknowledgment.

// Local shift state ‚Äî survives HXML re-fetch const shiftState = await AsyncStorage .getItem('active_shift'); // If server returns empty HXML but local // state shows clocked-in ‚Üí render clock-out if (shiftState?.status === 'clocked_in') { renderClockOutButton(shiftState); }
‚úÖ Clock-in persisted to AsyncStorage on device ‚Äî survives app restarts and HXML re-fetches
‚úÖ Server sync conflict resolution: local wins until server acknowledges clock-out
‚úÖ Connectivity status visible to worker ‚Äî "Saved on device" builds trust during offline periods
üìç Tap the connectivity strip in the phone to simulate losing connection ‚Äî clock-out button stays available
