Technical Root Cause Hypothesis

When a new build ships, the stored JWT or refresh_token is invalidated — either because the server revokes tokens tied to old app version headers, or the Keychain/SecureStorage key changes between builds (making the token unreachable), or the JWT payload structure changed and old tokens are rejected with no client-side migration. The loading screen hang indicates the app enters a token refresh loop with no timeout, no fallback to the login screen, and no cached data to display while offline. The fact that both iOS and Android fail identically confirms the failure is in shared JavaScript auth logic — not in platform-native code.

Technical Root Cause Hypothesis

The app makes concurrent API calls on load without timeout configuration optimized for cellular latency. On mobile data (higher latency, variable bandwidth), calls exceed timeout thresholds or fail silently. There is no offline-first data cache for core read-only data (schedule, roster, attendance). The fix involves implementing a write-through local cache (MMKV or WatermelonDB), adding request compression for cellular, and displaying cached data with a "last updated" timestamp when offline — syncing in background when connectivity is restored.
