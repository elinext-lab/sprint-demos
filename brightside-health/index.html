Section 2

What We Fix: The Auth Lifecycle

Three targeted engineering interventions. No architectural overhaul. 6–8 weeks.

The root cause is not a slow server. It is a mobile auth lifecycle that was designed for security (HIPAA-compliant token expiry) without a graceful recovery path for the highest-value user action: joining a scheduled therapy session.

FIX 1

Pre-Session Token Refresh

When the app is opened or resumed and an appointment exists within 30 minutes, silently attempt a token refresh using the stored refresh token before any UI is rendered. If refresh succeeds, proceed directly to session join. If refresh fails, present a specific, actionable error — not a loading spinner.

FIX 2

Video SDK Reconnect with Re-Auth

When a video media track drops during a session, the SDK reconnect logic must re-authenticate to the media server using the current session token. If the token has expired mid-session, the reconnect must trigger a background token refresh before re-establishing the encrypted media channel — preventing permanent video freeze.

FIX 3

Session Attempt Logging

Log every session join attempt — successful or not — with a timestamp, auth state, and failure reason (if applicable). This creates a clinical audit trail: "Patient attempted to join at T-2 min; technical failure." The care record reflects reality. Provider sees the attempt. The $95 is not charged. HITRUST controls are satisfied.

// Pre-session token refresh — fires on app resume when appointment is imminent async function prepareSessionToken(appointmentStartTime) { const minutesUntilSession = (appointmentStartTime - Date.now()) / 60000; if (minutesUntilSession <= 30 && minutesUntilSession > 0) { const tokenValid = checkTokenExpiry(); // local check, no network if (!tokenValid) { try { await silentTokenRefresh(); // ~400ms, no UI disruption logSessionAttempt({ status: 'token_refreshed', ts: Date.now() }); } catch (err) { // Do NOT show loading spinner — show specific, actionable recovery UI showSessionRecoveryUI(err); logSessionAttempt({ status: 'auth_failed', ts: Date.now(), error: err }); } } } }

Clinical Risk: Crisis Care Patients

Brightside's Crisis Care program uses the CAMS framework to manage patients at elevated suicide risk. Session continuity is clinically mandated. A technical failure that prevents a high-acuity patient from joining their scheduled session is not a UX issue — it is a care gap event. Under Brightside's HITRUST r2 certification, session management is an evaluated security control. Undocumented session failures accumulate audit risk. The fix closes the gap.

Summary

We have closed this exact failure pattern for other HIPAA-compliant telehealth platforms. The fix is scoped and isolated — no architectural overhaul required.
